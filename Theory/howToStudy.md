### 🚫 Coding immediately

If you code right after seeing the solution:

* The code is still in **working memory**
* You basically **copy from memory**
* It feels easy
* But **1–2 days later you forget**

So later you waste **more time relearning the same problem**.

### ✅ Waiting 20–30 minutes

When you wait:

* The exact steps fade
* You only remember the **core idea**
* When coding, your brain **reconstructs the logic**

That reconstruction is what builds **long-term memory**.

### ⏱ But you don't have to waste time

Instead of doing nothing, do this:

**Cycle method (very efficient):**

Problem A → watch solution
Problem B → watch solution
Problem C → watch solution

Now go back:

Code **Problem A**
Code **Problem B**
Code **Problem C**

You naturally get **30–40 min gap** without wasting time.

This is actually how many **competitive programmers train**.

Yes, **treat the brute force and optimal solution as the same problem**, not two separate problems.

### Why this matters

In interviews and contests, they usually expect this flow:

Problem → **Brute force idea** → **Optimization** → **Optimal solution**

If you only remember the optimal solution, sometimes you **don’t understand why it works**.

### How to practice it later

When you come back after the break:

1️⃣ First try to recall **brute force**

* What is the most obvious approach?
* What is its time complexity?

2️⃣ Then think:

* **Why is it slow?**
* What observation improves it?

3️⃣ Then code the **optimal solution**

You don’t always have to fully code brute force — sometimes just **explain the idea** is enough.

### Example structure in your head

When revisiting a problem:

This keeps the **reasoning chain**, which is what builds real problem-solving skill.
